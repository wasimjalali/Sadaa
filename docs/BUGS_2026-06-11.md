# Sadaa bug audit, 11 Jun 2026

Three reported problem areas: the Notes page (can't copy or edit notes), dictation
reliability, and Prompt Mode failing in terminals and chatbots. Each bug below was
traced to source and verified before fixing. Items marked "verified safe" were
investigated and found NOT to be bugs, so they were left alone.

## Notes page

The Notes page only supported add and delete. Everything the user asked for was
genuinely missing, not broken.

- **N1. Note text was not selectable.** `NotesPage.swift` rendered each note as a
  plain `Text`, and SwiftUI `Text` is not selectable without `.textSelection(.enabled)`.
  Home and History rows already use that modifier (`HomePage.swift:299`). You could
  not even click-drag to select a note.
  Fix: add `.textSelection(.enabled)` to the note text, matching Home/History.

- **N2. No Copy button.** The only control on a note row was the trash button. Home
  and History rows each have a morphing Copy button (`HomePage.swift:320-344`).
  Fix: add the same Copy button (with the `NSPasteboard.general` two-line helper the
  other pages use) plus the trash button.

- **N3. No way to edit a note.** `NotesStore` exposed only `all`, `add`, `remove`,
  and `SadaaViewModel` mirrored that. There was no `update` anywhere.
  Fix: add `NotesStore.update(id:text:)` (trim-and-reject-blank like `add`, find by
  id and save like `remove`), `SadaaViewModel.updateNote(_:text:)`, and an inline
  edit affordance on the note row. Editing keeps the note's `createdAt` and position.

## Dictation mode

- **D1. The `.delivering` state lasted ~0ms, so the busy mutex did not cover
  delivery.** `DictationController.process()` set `state = .delivering`, called the
  async `deliver(finalText)` (which posts Cmd-V, then verifies and restores the
  clipboard on a 0.25-0.6s timer), then immediately set `state = .idle`
  (`DictationController.swift:246-249`). The controller advertised `.idle` while a
  paste was still being verified, so a second hotkey tap could start a fresh
  recording, and its `startRecording` work, mid-delivery.
  Fix: thread the inserter's existing `completion` callback back into the controller
  so `.delivering` is held until delivery actually finishes. `TextInserter.deliver`
  already calls its completion on every path, so the mutex now spans the real
  delivery window.

- **D2. Voice edit swallowed every transcription error.** `VoiceEditController`
  transcribed the spoken instruction with `try?` and reported "Couldn't hear the
  instruction" for any failure, including a 401, a timeout, or no network
  (`VoiceEditController.swift:114-123`). The dictation path already does this
  correctly with a captured `lastError` and a real message. Voice edit had regressed
  from that, masking real provider failures (violates the fail-loud rule).
  Fix: mirror the dictation loop, capture the error, and surface a real message;
  keep "Couldn't hear the instruction" only for a genuinely empty transcript.

- **D3. Auto-stop could fire `toggle()` when not recording.** Both controllers wired
  `recorder.onAutoStop` straight to `toggle()` with no state guard
  (`DictationController.swift:79-81`, `VoiceEditController.swift:46-48`). A late
  auto-stop dispatched to the main queue after the state had already left
  `.recording` could mis-toggle (worst case, start a new recording from `.idle`).
  Fix: `guard state == .recording` before the auto-stop toggle, in both controllers.

- **Verified safe (not changed): empty-text delivery.** Both formatter parsers
  already guard against empty output (`parse` falls back to the raw transcript,
  `parseOptimized` throws and the pipeline falls back to raw), and the raw transcript
  is guaranteed non-empty by the no-speech guard, so `deliver("")` is not reachable.
- **Verified safe (not changed): the consumption-verified paste / clipboard-restore
  logic.** This is deliberate, hard-won code with a documented history. The
  before/after growth check has edge cases on AX-churning fields, but touching it
  risks reintroducing the clipboard-loss bug, so it was left as-is.

## Prompt Mode

- **P1. Browser-based chatbots were never targeted.** Prompt Mode only runs when the
  frontmost app's bundle id is in `promptModeApps`, whose default was the code/terminal
  apps plus the Claude and ChatGPT desktop apps (`AppSettings.swift:180-185`). No web
  browsers, so dictating into ChatGPT/Claude/Gemini in Safari, Chrome, Arc, etc. fell
  through to plain smart formatting with no optimization. This is the "doesn't work in
  other chatbots" report.
  Fix: add the common browsers and the popular non-Apple terminals (Ghostty, Kitty,
  Alacritty, WezTerm, Hyper) to the default `promptModeApps`. Kept separate from
  `FormattingProfiles.code.bundleIDs` so browsers don't inherit the code formatting
  profile. Users can still edit the list in Settings.

- **P2/P3. Prompt Mode failed silently when it could not run.** When Prompt Mode was
  enabled in a targeted app but had no GPT deployment configured, or Smart formatting
  (the master switch) was off, the dictation was delivered raw with no explanation
  (`AppDelegate.swift:202-206`). This is the "doesn't work in the terminal and I don't
  know why" report.
  Fix: restructure the format closure so an enabled-but-can't-run Prompt Mode surfaces
  a clear notice ("Prompt mode needs a GPT deployment in Settings." /
  "Prompt mode needs Smart formatting on.") via the existing `pendingDeliveryNotice`
  path. Smart formatting stays the master switch (no GPT runs when it is off); the
  only change is that the silent no-op is now visible.

- **P4. A non-conforming optimizer reply was discarded.** `parseOptimized` decodes the
  model's content strictly as `{text, newTerms}` and throws on anything else
  (`AzureChatFormatter.swift:276-298`). A model that wraps the JSON in a markdown code
  fence (```json ... ```) despite the instruction would throw, and the pipeline would
  paste the raw dictation instead of the optimized prompt.
  Fix: strip a leading/trailing markdown code fence before decoding, in both
  `parseOptimized` and `parse`.

## Round 2 (same day): HUD waveform + provider errors

### Recording HUD logo

- **The mark pulsed slowly instead of flowing like a wave.** All five bars
  scaled one shared `level` value (`HUDView.swift` `SadaaWaveBars`), so the logo
  rose and fell in lockstep, and it only repainted when a new level sample
  arrived (10Hz) through a 0.12s ease-out, which read as sluggish.
  Fix: drive the bars from a continuous time source (`TimelineView(.animation)`)
  so they ripple every frame, with a per-bar phase offset so the five bars travel
  as a wave. The mic level sets the wave's amplitude and speed (calm when quiet,
  lively when loud); a faint idle ripple keeps it visibly "listening". Reduced
  Motion falls back to amplitude-only. Also raised the level push from 10Hz to
  30Hz so the amplitude envelope tracks speech promptly, and dropped the unused
  recording-seconds counter.

### HTTP 400 / 500 provider errors

Traced the request path end-to-end (multipart body, WAV header, URL/api-version,
headers, response_format). The request building is correct. Conclusion:
- **500 is a genuine provider error** (transient Azure/OpenAI capacity), handled
  by the existing fallback chain.
- **400 was most often self-inflicted:** a header-only or near-empty WAV (instant
  tap, denied mic, capture race) was uploaded and the provider rejected it with a
  400 that read as a mysterious provider failure.
- **The provider's response body was being discarded**, so a genuine error and a
  bug both showed the same opaque "HTTP 400 from provider" - impossible to tell
  apart.

Fixes:
- Surface the provider's own error text in the message ("HTTP 400: audio file is
  too short", "HTTP 429: insufficient_quota", etc.) in both the dictation and
  voice-edit error paths, so genuine-vs-bug is now obvious.
- Guard near-empty audio (< ~100ms) before the upload and bill, with a plain
  "Recording was too short." notice instead of a provider 400.
